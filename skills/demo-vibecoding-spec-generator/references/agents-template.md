```md
# AGENTS

## 开发流程
1. 先确认输入文档、目标范围、输出物和验收标准，再开始拆分或编码。
2. 先产出或更新 `specs/*.md`，再按依赖顺序实施。
3. 每个任务必须写清输入、输出、依赖、实现边界和验收标准。
4. 变更后同步更新相关文档、测试和发布说明。

## 代码规范
1. 遵循项目既有技术栈和目录结构，避免跨层直接依赖。
2. 保持命名清晰、函数职责单一、错误处理完整、日志可追踪。
3. 新增或修改行为时补齐必要测试，避免无验证提交。
4. 不在无明确理由时引入大体量抽象、重复封装或隐式副作用。

## 接口调用标准
1. 所有接口必须定义请求参数、响应结构、错误码和超时策略。
2. 外部调用必须具备超时、重试或降级策略，并记录关键失败日志。
3. Webhook/回调接口必须做签名校验、来源校验和幂等控制。
4. 响应与日志中不得泄露密钥、令牌或敏感个人信息。

## 数据存储与数据库操作规范
1. 明确读写边界，禁止在业务流程中隐式写库或隐式修改共享状态。
2. 结构变更必须带迁移与回退方案。
3. 写操作应具备审计信息；使用 JSON 或内存存储时也要遵守相同校验规则。
4. 删除、覆盖和批量更新操作必须有保护条件和回退办法。

## 团队协作机制
1. 以 `specs/*.md` 作为任务拆分和协作基线。
2. 合并前至少完成一次代码评审，并确认依赖任务已就绪。
3. 重大技术决策记录到 `docs/` 或对应 spec 中，避免只留在聊天记录里。
4. 发布前确认测试、回退方案和关键观测项；发布后核对核心日志与指标。
```
