# AGENTS

## 开发流程
1. 需求、范围、验收标准不清时先补齐上下文，不直接写代码。
2. 文档冲突时以 `docs/需求规格说明书.md` 为准，架构、数据库、API 文档只用于补充实现边界。
3. 先维护 `specs/*.spec.md`，再按依赖顺序编码；每次只推进一个可运行切片。
4. 每日都要有可验证结果，优先保证聊天主链路和定时推送链路跑通。
5. 需求变更、取舍结论和延期项必须同步写回 spec，不只保留在对话里。

## 代码规范
1. 维持 Demo 的精简架构：TypeScript + Express + `@line/bot-sdk` + LangChain + `node-cron` + `axios` + `dotenv`。
2. 优先复用成熟框架能力，不为了 Demo 额外引入数据库、中间件平台或复杂抽象。
3. 函数保持单一职责，模块边界清晰，避免跨层直接依赖和重复封装。
4. 外部调用、关键分支和异常路径必须有错误处理与可读日志。
5. 未经 spec 明确批准，不扩展高成本优化项。

## 接口调用标准
1. `POST /webhook` 按 LINE 回调约定处理，必须接入签名校验中间件。
2. 内部管理接口统一返回 JSON，并明确成功结构、错误码和失败原因。
3. 外部 API 调用必须配置超时；重试和降级只做 Demo 必要级别。
4. 不在响应体和日志中暴露密钥、令牌或完整敏感配置。

## 数据存储与数据库操作规范
1. Demo 阶段不引入正式数据库，默认使用内存 `Map`、`BufferMemory` 和 `src/config/tasks.json`。
2. 对话上下文按 `userId` 分组管理，并限制保留最近 3 轮对话。
3. 定时任务配置以静态 JSON 为准，不实现在线增删改全套能力。
4. 如果未来要引入 PostgreSQL 或 Redis，必须先补 spec，再实施迁移。

## 团队协作机制
1. 一个 `*.spec.md` 对应一个可交付模块，不把多个模块混到同一次实现里。
2. 迭代顺序固定为：启动与 LINE 链路 -> LLM 上下文 -> 定时推送 -> 治理与收尾。
3. 每次提交都要说明当前完成了哪个 spec、阻塞点是什么、下一步接哪一个 spec。
4. Demo 阶段允许暂缓性能优化和完整安全治理，但必须显式记录延期项。
